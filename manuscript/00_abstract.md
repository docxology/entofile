# Abstract {#sec:abstract}

ENTO — an **EN**crypted, **T**yped, **O**mnitrack container — is a flat ZIP archive [@zipappnote2024] for bundling heterogeneous research artifacts (time series, genomics slices, spectrograms, provenance proofs) into one verifiable file. It combines typed *omnitrack* manifests, per-track authenticated encryption with AES-256-GCM on format {{FORMAT_VERSION}} (suite {{CRYPTO_BACKEND_DEFAULT}}) grounded in AES, AEAD, and GCM standards and security analysis [@nistfips197; @rfc5116; @dworkin2007gcm; @mcgrew2004gcm], {{COUNT_OBSERVABILITY_LEVELS}} graded observability levels that control how much manifest metadata a recipient sees, and an optional minimal hash-chained proof export aligned with PROV-style derivation and tamper-evident log lineage [@merkle1988digital; @haber1991timestamp; @w3cprov2013; @w3cprovo2013]. The reference implementation lives under `src/` with benchmarks orchestrated by `scripts/ento_analysis.py`; release facts and measured values are injected from pipeline outputs rather than hand-edited prose, following reproducible-computing practice around scripted analyses, versioned inputs, and accessible run artifacts [@sandve2013reproducible; @wilson2017goodenough].

Release {{PAPER_VERSION}} documents default ENTO wire format {{FORMAT_VERSION}}. The default writer emits {{FORMAT_VERSION}}; the prior supported formats ({{FORMAT_VERSIONS_COMPATIBILITY}}) remain explicit compatibility choices rather than disappearing from the decrypt path. The default profile adds {{FORMAT_NEXT_BINDING_DESCRIPTION}} without changing the ZIP or binary track layout.

Compared with array stores (HDF5 [@hdf5_tr], Zarr [@zarr_v2]) and publication containers (EPUB [@epub31]), ENTO emphasizes authenticated track envelopes and graded manifest export rather than chunk-level partial I/O or rendering semantics. Preservation frameworks such as OAIS and PREMIS define archive responsibilities and preservation metadata [@ccsds2024oais; @premis2015], while RO-Crate [@rocrate2024; @soilandreyes2022rocrate] and BagIt [@bagit2018] provide complementary research-object and transfer packaging. ENTO occupies the narrower file-format layer: it specifies ciphertext layout in `data/ento_track_header.ksy` (nonce({{NONCE_BYTES}}) || tag({{TAG_BYTES}}) || ciphertext) using a Kaitai Struct definition [@kaitai2026] and validates manifests with JSON Schema [@jsonschema2020]. Verification deliberately separates key-authenticated integrity from keyless corruption detection, and {{COUNT_COMPATIBILITY_FORMATS}} compatibility formats remain version-dispatched alongside the default. The default profile includes associated-data binding [@rfc5116], PADMÉ length padding [@nikitin2019purb], and canonical exported-manifest context through {{FORMAT_NEXT_AAD_TEMPLATE}}, bounding the on-disk ciphertext member-size channel to a coarse bucket. Length is never hidden exactly: at non-sealed observability levels the manifest's cleartext `byte_length` discloses it directly, and even under sealed export the padded member size still reveals the PADMÉ bucket.
